### **The Complementary Value of Semantic and Numerical Representations in Clinical Risk Prediction**

### **1\. Overarching Research Question**

Does a general-purpose Large Language Model's semantic representation of *aggregate clinical data* offer predictive information that is complementary or superior to a traditional numerical representation of the *exact same data*? Furthermore, how sensitive is this semantic representation to the specific format and prompting strategy used during encoding?

### **2\. Core Hypotheses**

This project will test three primary, interlinked hypotheses, with a fourth exploratory one.

* **H1 (The "Information-Equivalent Representation" Hypothesis):** Given an identical set of aggregate clinical features, a semantic representation generated by a State-of-the-Art (SOTA) text embedding model will outperform a strong numerical baseline (XGBoost) trained on the raw numerical vector. This isolates the effect of the representation modality (semantic vs. numerical).  
* **H2 (The "Complementary Signal" Hypothesis):** The semantic embeddings from the LLM and the numerical features capture independent, non-redundant predictive signals from the same source information. Therefore, a hybrid Mixture of Experts (MoE) model combining both will significantly outperform either model in isolation.  
* **H3 (The "Encoding Sensitivity" Hypothesis):** The performance of the semantic representation is highly sensitive to the method of encoding. Specifically, different serialization formats (e.g., Markdown vs. JSON vs. Narrative) and prompt engineering strategies (e.g., minimalist vs. persona-driven) will yield embeddings of varying quality, resulting in statistically different downstream prediction performance.  
* **Exploratory Hypothesis (Domain Specificity):** A domain-specific EHR foundation model (CLMBR) may offer different predictive capabilities compared to a general-purpose text embedding model, although a direct comparison is confounded by differences in required data formats (event-sequence vs. aggregate).

### **3\. Datasets and Prediction Task**

* **Primary Dataset:** MIMIC-Extract dataset.  
* **Prediction Target:** In-hospital Mortality (mort\_hosp). A well-defined, clinically relevant binary classification task.

### **4\. Experimental Arms (The Models)**

We will define and train models to systematically test these hypotheses.

* **Model 1: The Numerical Expert (XGBoost Baseline)**  
  * **Input Data:** A numerical vector of aggregate features (mean, std, min, max) for the 104 MIMIC-Extract variables over the first 24 hours.  
  * **Purpose:** To establish a strong, purely numerical baseline that is **informationally-equivalent** to the semantic models. This is the control for H1.  
* **Model 2: The Semantic Expert (LLM Embedding Suite)**  
  * This is not a single model, but a suite of models designed to test H3. The core information remains the aggregate features from Model 1\.  
  * **Input Data Variations:**  
    * **Format Sub-analysis:** The aggregate data will be serialized into multiple formats (e.g., **2a: Markdown**, **2b: JSON**, **2c: Narrative Prose**).  
    * **Prompt Sub-analysis:** For a fixed format (e.g., Markdown), we will test different instructions (e.g., **2d: Minimalist Prompt** with no instructions, vs. **2e: Sophisticated Prompt** with a defined persona and task).  
  * **Embedding Model:** Google's text-embedding-004 (or latest equivalent).  
  * **Prediction Head:** A simple Logistic Regression classifier trained on the resulting embeddings.  
  * **Purpose:** To test the "Encoding Sensitivity" hypothesis (H3). The best-performing variant from this suite will become the "champion" Semantic Expert for testing H1 and H2.  
* **Model 3: The Hybrid Expert (Mixture of Experts)**  
  * **Architecture:** A meta-model (Logistic Regression) trained on the concatenated predictions (probabilities) from Model 1 and the **champion** variant of Model 2\.  
  * **Input:** For a given patient, it takes the input \[prob\_xgboost, prob\_llm\_champion\].  
  * **Purpose:** To directly test the "Complementary Signal" hypothesis (H2).  
* **Model 4 (Exploratory): The Domain-Specific Expert (CLMBR Embedding)**  
  * **Input Data:** Patient data formatted into the event-sequence format required by the EHRSHOT benchmark's CLMBR model.  
  * **Prediction Head:** A Logistic Regression classifier, identical to Model 2's, for fair comparison of the classifier.  
  * **Purpose:** To explore the Domain Specificity hypothesis (Exploratory). Results will be interpreted with caution due to the confounding variable of data format.

### **5\. Step-by-Step Analysis Workflow**

**Phase I: Data Preparation & Baseline Establishment (Weeks 1-2)**

1. **Feature Engineering:** Process the MIMIC-Extract dataset to create the definitive **aggregate feature set** (mean, std, min, max for 104 variables) for the first 24 hours of ICU stay. This is your single source of truth for information.  
2. **Data Partitioning:** Use the fixed train/validation/test splits from Wang et al. (2020) and Gao et al. (2024) for all models to ensure valid comparisons.  
3. **Train Model 1 (XGBoost):** Train and tune the XGBoost model on the aggregate numerical data. This establishes your primary performance benchmark.

**Phase II: Semantic Encoding Analysis (Testing H3) (Weeks 3-5)**

1. **Serialization Design:** Create the templates for each serialization format (Markdown, JSON, Narrative).  
2. **Prompt Design:** Craft the different system prompts (Minimalist vs. Sophisticated).  
3. **Generate Embeddings:** For every patient, generate a complete set of embeddings for each variation (2a, 2b, 2c, 2d, 2e, etc.).  
4. **Train & Evaluate Semantic Variants:** Train a separate Logistic Regression head for each embedding set.  
5. **Evaluate H3:** Directly compare the AUROC of all Model 2 variants. A statistically significant difference between them will provide evidence for H3.  
6. **Select Champion:** Identify the single best-performing semantic representation. This "champion" version of Model 2 will be used for all subsequent comparisons.

**Phase III: Core Hypothesis Testing (H1 & H2) (Weeks 6-7)**

1. **Evaluate H1:** Directly compare the AUROC and AUPRC of the champion Model 2 against Model 1\. This is a clean test of representation modality.  
2. **Train Model 3 (Mixture of Experts):** Generate out-of-fold predictions from Model 1 and the champion Model 2\. Use these predictions to train the meta-Logistic Regression model.  
3. **Evaluate H2:** Compare the AUROC of Model 3 against both Model 1 and the champion Model 2\. A statistically significant improvement is the primary endpoint.

**Phase IV: Final Analysis, Interpretation, and Reporting (Weeks 8-10)**

1. **Primary Performance Analysis:** Compile a master table comparing key metrics (AUROC, AUPRC, etc.) for Model 1, the full suite of Model 2 variants, and Model 3\.  
2. **Complementarity Analysis:**  
   * Calculate the correlation between the prediction errors of Model 1 and the champion Model 2\. Low correlation is strong evidence for H2.  
   * Qualitatively analyze cases where the models disagree to understand *why* their signals are complementary.  
3. **Exploratory Analysis (Time Permitting):**  
   * If time and resources allow, execute the workflow for Model 4 (CLMBR).  
   * Compare its performance to the champion Model 2, explicitly noting in the write-up that the comparison is confounded by data format.  
4. **Write-up:** Structure the final report around the three core hypotheses. Begin by presenting the results of the encoding analysis (H3), which justifies the choice of the champion model. Then, use this champion model to present clear, evidence-backed answers for H1 and H2. Discuss the exploratory findings last.
